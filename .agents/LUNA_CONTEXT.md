## 1. 存在意義

### Mission（開発判断の基準）

**Luna コミュニティにつながるユーザーを集め、持続的なマッチングエコシステムを構築する。**

> **会社としてのミッション:** 安心して "自分らしさ" を追求できる社会の創造

- Luna は欲望を肯定する場所ではあるが、欲望を免罪しない
- Luna が扱うのは快楽ではない。**関係性**である

### Core Principles

1. **ユーザー信頼が最優先** - 信頼を損なう施策はどんなに数字が良くても NoGo
2. **安全性は非交渉** - 安全に関わる妥協は一切しない
3. **持続可能な成長** - 短期のハック的施策より、長期の仕組み作りを優先
4. **データで判断** - 「なんとなく」ではなく、計測可能な指標で意思決定

Luna は短期 KPI よりも以下を優先する:

1. **女性の安心**
2. **コミュニティの治安**
3. **継続性**

治安が崩れれば、市場は崩壊する。

## 4. 市場創造思想

### 企業の目的は顧客の創造

Luna は既存 SM ユーザーを奪わない。
「成熟した SM 関係性」を志向する顧客を **創る**。これが市場創造。

### Blue Ocean

| 既存市場 | Luna の市場 |
|---------|-----------|
| 刺激 × 消費 | 安心 × 関係性 |
| 衝動 × 即時性 | 成熟 × 長期継続 |

### Luna が破壊する前提

| 既存の前提 | Luna の前提 |
|-----------|-----------|
| SM は危険 | SM は合意と対話の技術 |
| 欲望は免罪される | 欲望は責任を伴う |
| マッチは早いほど良い | 関係性には適切な速度がある |
| 売上＝刺激 | 売上＝信頼 × 継続 |

### 独占すべき概念

- **SM は関係性の技術**
- **成熟する意思**
- **治安は資産**

## 10. サービス構成

Luna は2つの独立したサービスで構成される。**データは絶対に混ぜない。**

| サービス | 概要 | リポジトリ |
|---------|------|-----------|
| **Luna** | マッチングアプリ本体 | Luna-company org（参照のみ） |
| **Utata** | Luna 関連 SNS | Luna-company org（参照のみ） |

- 分析・開発時は必ずどちらのサービスか明示すること
- Luna のデータで Utata の施策を判断しない（逆も同様）

---

## 11. 本番環境ルール

### データベース

- **READ ONLY のみ**。書き込み権限は誰も持たない
- Luna DB 参照: Redash 経由を優先
- Utata DB 参照: Metabase 経由（VPN 必須）
- 直接 SQL 実行が必要な場合は、ReadOnly レプリカのみ使用

### リポジトリ

- **Luna-company org は参照のみ**。push / commit / ファイル編集は禁止
- 開発は hinominant org で行う
- Luna-company のコードを hinominant にコピーして公開しない

### 決済

- **Credix** を使用（Stripe ではない）
- 決済に関わる変更は必ずオーナーに確認してから実施

---

## 12. 開発ガードレール

### 絶対禁止（L4: 即時停止）

- 本番 DB への書き込み
- Luna-company org への push
- 本番サービスに影響を与える操作
- `.env` / 認証情報の git commit
- お金がかかる操作（オーナー確認なし）

### 分析ルール（p-hacking 防止）

- 多重検定には Bonferroni 補正を適用
- サブグループの後出し選択（cherry-picking）禁止
- 帰無結果も正直に報告する
- 片側検定への事後切り替え禁止
