# オーケストレーション設計テンプレート

> **使い方**: 3つの役割（整理役・批判役・実行役）に順番にプロンプトを投げ、それぞれの回答を記録してください。
> 記入例を参考に、空欄版を埋めてからClaude Codeに渡すプロンプトテンプレートで一気にコードを生成します。

---

## 記入例: 「社内FAQ検索ツール」

### 1. 整理役（Organizer）

**プロンプト:**

```
あなたはサービス設計の整理役です。
以下の情報を整理してください。

サービス名: 社内FAQ検索ツール
ターゲットユーザー: 新入社員（入社1年目）
解決したい課題: 「誰に聞けばいいか分からない」「同じ質問が何度も来る」

以下の3点を整理してください:
1. このサービスの目的（1文で）
2. ターゲットユーザーの具体的なペルソナ（名前・部署・困りごと）
3. 提供する価値（3つ）
```

**期待する回答例:**

```
1. 目的:
   新入社員が社内の手続き・ルール・ノウハウを自力で素早く見つけられるようにする。

2. ペルソナ:
   田中 太郎（23歳）/ 営業部 / 入社3ヶ月目
   - 経費精算の方法が分からないが、先輩が忙しそうで聞けない
   - Slackで質問したが、どのチャンネルが正しいか分からず放置された
   - 社内Wikiがあるらしいが、検索しても欲しい情報が出てこない

3. 提供する価値:
   - 24時間いつでも質問できる安心感
   - キーワードだけで関連FAQが見つかる簡単さ
   - 「よくある質問」ランキングで先回りして不安を解消
```

---

### 2. 批判役（Critic）

**プロンプト:**

```
あなたはサービス設計の批判役です。
厳しい目で弱点を指摘してください。

以下の設計内容を確認してください:
- サービス名: 社内FAQ検索ツール
- 目的: 新入社員が社内の手続き・ルール・ノウハウを自力で素早く見つけられるようにする
- ターゲット: 新入社員（入社1年目、営業部）
- 提供価値: 24時間質問可能、キーワード検索、よくある質問ランキング

以下の観点で弱点を指摘してください:
1. ターゲットの定義に穴はないか
2. 「本当にこれで課題が解決するか」の疑問点
3. 最も失敗しやすいポイント
4. 改善提案（具体的に）
```

**期待する回答例:**

```
1. ターゲットの穴:
   - 新入社員だけでなく、異動者・中途入社者も同じ課題を抱えている
   - 営業部に限定しているが、部署ごとにFAQの内容は大きく異なる
   → 改善: 「社歴6ヶ月未満の全社員」に広げる方が投資対効果が高い

2. 本当に解決するかの疑問:
   - FAQ検索ツールがあっても、そもそもFAQデータが整備されていなければ意味がない
   - 「検索して出てこない」体験が1回でもあると、以後使われなくなる
   → 改善: FAQ登録フローの設計が先。最低100件のFAQを事前に用意してからリリース

3. 最も失敗しやすいポイント:
   - 運用が続かない。FAQ更新の担当者が決まっておらず、情報が古くなる
   → 改善: 「月1回のFAQ棚卸し」をルール化し、各部署の担当者を明記する

4. 改善提案:
   - 「質問したけど見つからなかった」ログを自動記録し、FAQ追加候補にする
   - 利用率ダッシュボードを管理者向けに用意する
```

---

### 3. 実行役（Executor）

**プロンプト:**

```
あなたはサービス設計の実行役です。
MVP（最小限の製品）の機能を定義してください。

整理役の設計:
- 目的: 新入社員が社内の手続き・ルール・ノウハウを自力で素早く見つけられるようにする
- ターゲット: 社歴6ヶ月未満の全社員
- 提供価値: 24時間質問可能、キーワード検索、よくある質問ランキング

批判役の指摘:
- FAQデータ100件の事前準備が必要
- 「見つからなかった」ログの記録
- FAQ更新の運用ルール化

以下を定義してください:
1. MVP機能一覧（5つ以内、優先順位付き）
2. 各機能の1行説明
3. MVPに入れない機能と、その理由
4. 画面構成（ページ一覧）
```

**期待する回答例:**

```
1. MVP機能一覧（優先順位順）:
   P1: FAQ検索 - キーワード入力でFAQ一覧を表示
   P2: FAQ詳細表示 - 選択したFAQの本文を表示
   P3: よくある質問TOP10 - 閲覧数順のランキング表示
   P4: カテゴリ絞り込み - 部署・手続き種別で絞り込み
   P5: 「見つからない」ボタン - 検索結果が役に立たなかった時の記録

2. MVPに入れない機能:
   - FAQ管理画面（管理者向け） → CSVインポートで代替
   - チャットbot連携 → MVP検証後に追加
   - 利用率ダッシュボード → スプレッドシートで手動集計

3. 画面構成:
   - トップページ（検索バー + ランキング）
   - 検索結果一覧ページ
   - FAQ詳細ページ
   - カテゴリ一覧ページ
```

## Claude Codeに渡すプロンプトテンプレート

3役の設計が完了したら、以下のプロンプトをClaude Codeに貼り付けてコードを生成します。

### プロンプト:

```
以下の設計書に基づいて、Next.js（Static Export）のWebサイトを作成してください。

## サービス概要
- サービス名: [あなたのサービス名]
- 目的: [整理役の回答]
- ターゲット: [批判役の指摘を反映したターゲット]

## MVP機能
[実行役の回答をそのまま貼り付け]

## 画面構成
[実行役の回答から画面構成を貼り付け]

## 技術要件
- Next.js + Tailwind CSS
- Static Export（`output: 'export'`）
- DB無し、API無し、フロントエンドのみ
- レスポンシブデザイン
- 日本語UI

## 制約
- `npm run build` でエラーが出ないこと
- ページ遷移はNext.jsのLinkを使うこと
- 画像は使わず、テキストとアイコン（絵文字）で構成すること

上記を満たすコードを生成してください。
ファイル構成も提案してください。
```

---

## Tips

- **整理役の回答が曖昧だと、後工程がすべて曖昧になる。** 「1文で」「3つ」など数を指定するのがコツ。
- **批判役は遠慮させない。** 「厳しい目で」「弱点を」と明記する。
- **実行役には必ず「5つ以内」と制限をかける。** 制限がないとMVPが膨れ上がる。
- **3役の回答は必ず記録してからClaude Codeに渡す。** 途中で何を決めたか分からなくなるのを防ぐ。
