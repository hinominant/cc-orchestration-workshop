### M9: STEP別理解度自己評価（段階的開示の効果測定）

**目的:** 段階的開示（構造を見せる → 動きを見せる → 体験する → 改変する）の各段階で理解度がどう変化するかを測定する。

**計測方法:** 各STEPの終了時に参加者が5段階で自己評価する。Googleフォーム（リアルタイム回答）または挙手方式（簡易版）。

**設問（各STEP共通）:**

> 「今のSTEPの内容について、自分の理解度を5段階で教えてください。」
> 1: 全く分からなかった → 5: 十分理解できた

**記録テンプレート:**

| STEP | 段階的開示フェーズ | 参加者A | 参加者B | 参加者C | 平均 |
|------|----------------|--------|--------|--------|------|
| STEP0 | - | /5 | /5 | /5 | /5 |
| STEP1 | 構造を見せる | /5 | /5 | /5 | /5 |
| STEP2 | 動きを見せる | /5 | /5 | /5 | /5 |
| STEP3 | 体験する | /5 | /5 | /5 | /5 |
| STEP4 | 指示を変えてみる | /5 | /5 | /5 | /5 |
| STEP5 | - | /5 | /5 | /5 | /5 |
| STEP6 | - | /5 | /5 | /5 | /5 |
| STEP7 | 全体を理解する | /5 | /5 | /5 | /5 |

**分析観点:**

| 観点 | 期待される結果 | アラート条件 |
|------|-------------|------------|
| STEP1→STEP2の上昇 | デモを見て理解が進む（+0.5以上） | 変化なし or 低下 → デモの見せ方に問題 |
| STEP2→STEP3の上昇 | 体験して理解が深まる（+0.5以上） | 低下 → 手順が分かりにくい |
| STEP3→STEP4の維持or上昇 | カスタマイズで理解が定着 | 低下 → カスタマイズの難易度が高すぎる |
| 全STEP平均 | 3.5以上 | 3.0未満 → カリキュラム全体の見直しが必要 |

## B. 失敗時の組織学習設計

### 複数ゴール定義

| ゴール | 条件 | 到達判定 | 対象STEP |
|-------|------|---------|---------|
| 最低限ゴール | CLAUDE.mdとエージェントの役割を理解できた | M7・M8の事後確認で核心を含む回答がある | STEP1-2 |
| 標準ゴール | ローカルでWebページが表示された | `npm run dev` で `localhost:3000` にページが表示される | STEP3まで |
| 理想ゴール | 公開URLを取得できた | Vercelの公開URLでページが表示される | STEP6まで |

**ゴール切り替え判断基準:**

| 状況 | 判断 | アクション |
|------|------|----------|
| STEP3終了時点で残り30分以上 | 理想ゴールを目指す | 通常通りSTEP4-6へ |
| STEP3終了時点で残り20-30分 | 標準ゴール確定 | STEP4を簡略化、STEP5-6へ |
| STEP3が残り30分で未完了 | 最低限ゴールに切り替え | 自動ビルドを中断し、講師の完成版を見せる |
| STEP1が残り80分で未完了 | 最低限ゴール（概念理解のみ） | 講師デモに集中し、構造理解を確実にする |

---

### 「失敗」の定義と対処フロー

#### パターン1: 環境トラブル

**定義:** ソフトウェアのインストール・設定の問題でSTEP0が完了しない

**対処フロー:**
1. 講師が横で原因を特定する（最大5分）
2. 5分で解決しない場合 → スターターリポジトリの再Fork を案内する
3. 再Forkでも動かない場合 → 講師のPCで一緒に操作する（ペアワーク形式）
4. 全員が環境トラブル → 講師のPC画面を映して全員でモブ形式で進める

**フォールバック案内テンプレート:**
```
環境が動かない場合、スターターリポジトリを再度forkしてください。
https://github.com/hinominant/cc-workshop-starter

1. 上記URLを開く
2. 右上の「Fork」ボタンを押す
3. 自分のアカウントにforkされる
4. Claude Codeでfork先のリポジトリを開く
```

---

#### パターン2: ビルドエラー

**定義:** エージェントが生成したコードで `npm run dev` または `npm run build` でエラーが発生する

**対処フロー:**
1. エラーメッセージをClaude Codeに貼り付けて修正を依頼する（1回目）
2. 修正されたら再ビルドする
3. 2回修正しても解決しない場合 → 講師が直接エラーを確認して修正支援する（最大3分）
4. 3分で解決しない場合 → フォールバックリポジトリに切り替える

---

#### パターン3: 時間不足

**定義:** 予定のSTEPが時間内に完了しない

**対処フロー:**
1. 残り時間を確認し、到達可能なゴールを判断する（上記ゴール切り替え判断基準を参照）
2. カスタマイズの範囲を縮小（色変更1回のみ）
3. Git/GitHub操作を講師が代行して時間を節約する
4. 最低限ゴールに切り替え、概念理解を確実にする

**簡略化オプション:**

| 項目 | 通常 | 簡略化 |
|------|------|--------|
| ページ数 | 複数ページ | 1ページ（トップのみ） |
| カスタマイズ | 複数回の修正依頼 | 色変更1回のみ |
| Git操作 | 参加者が自分で実行 | 講師が代行 |
| Vercel設定 | 参加者が自分で設定 | 講師が画面共有で代行 |

### リスクと緩和策

| リスク | 発生確率 | 影響度 | 緩和策 |
|-------|---------|-------|-------|
| 参加者の環境構築が間に合わない | 高 | 高 | MS4で2日前にフォロー、スターターリポジトリの再Fork手順を用意 |
| リハーサルで120分を超過する | 中 | 高 | 各STEPのバッファ時間を設定、簡略化オプションを用意 |
| エージェントが想定外の動きをする | 中 | 高 | フォールバック用の完成版リポジトリを事前デプロイ |
| 当日のネットワーク障害 | 低 | 高 | テザリング用モバイル端末を用意 |
| Claude Code APIの障害 | 低 | 高 | フォールバックリポジトリで対応、手動コーディングは行わない |
| 参加者のスキル差が大きい | 中 | 中 | 事前アンケートでレベル把握、L3の参加者にはペアサポート役をお願い |
| アンケート回収率が低い | 低 | 中 | 講座終了直後にその場で記入してもらう（退室前に完了） |
