### M1: 事前スキルレベル（Googleフォーム）

**設問:**

| # | 質問 | 回答形式 |
|---|------|---------|
| Q1 | AIツール（ChatGPT、Claude等）を使ったことがありますか？ | はい / いいえ |
| Q2 | Q1で「はい」の場合、頻度はどのくらいですか？ | 毎日 / 週数回 / 月数回 / 数回だけ |
| Q3 | プログラミングの経験はありますか？ | ある（業務で使用）/ ある（趣味・学習）/ なし |
| Q4 | ターミナル（コマンドプロンプト）を使ったことがありますか？ | はい / いいえ |
| Q5 | GitHubのアカウントを持っていますか？ | はい / いいえ |
| Q6 | 今回の講座で最も期待していることは何ですか？ | 自由記述 |

**スキルレベル分類（Q1-Q5の回答に基づく）:**

| レベル | 条件 | 対応方針 |
|-------|------|---------|
| L1: 完全初心者 | AI未使用 + プログラミング未経験 | 手順を一つずつ確認しながら進める |
| L2: AI経験あり | AI使用経験あり + プログラミング未経験 | AI操作は省略可、コード周辺を重点サポート |
| L3: 技術経験あり | プログラミング経験あり | 自走を促し、応用的な質問に時間を使う |

## B. 失敗時の組織学習設計

### 複数ゴール定義

| ゴール | 条件 | 到達判定 | 対象STEP |
|-------|------|---------|---------|
| 最低限ゴール | 3役分業の概念を理解できた | Q6の自由記述で「役割を分ける」趣旨の回答がある | STEP1-2 |
| 標準ゴール | ローカルでWebページが表示された | `npm run dev` で `localhost:3000` にページが表示される | STEP4まで |
| 理想ゴール | 公開URLを取得できた | Vercelの公開URLでページが表示される | STEP6まで |

**ゴール切り替え判断基準:**

| 状況 | 判断 | アクション |
|------|------|----------|
| STEP4終了時点で残り25分以上 | 理想ゴールを目指す | 通常通りSTEP5-6へ |
| STEP4終了時点で残り15-25分 | 標準ゴール確定 | STEP5のみ実施、STEP6は講師が口頭で説明 |
| STEP4が残り20分で未完了 | 最低限ゴールに切り替え | コード生成を中断し、講師の完成版を見せる |
| STEP3が残り30分で未完了 | 最低限ゴール（設計体験のみ） | 3役テンプレの記入を完了させることに集中 |

---

### 「失敗」の定義と対処フロー

#### パターン1: 環境トラブル

**定義:** ソフトウェアのインストール・設定の問題でSTEP0が完了しない

**対処フロー:**
1. 講師が横で原因を特定する（最大5分）
2. 5分で解決しない場合 → fork用フォールバックリポジトリを案内する
3. フォールバックでも動かない場合 → 講師のPCで一緒に操作する（ペアワーク形式）
4. 全員が環境トラブル → 講師のPC画面を映して全員でモブ形式で進める

**フォールバックリポジトリ案内テンプレート:**
```
環境が動かない場合、以下のリポジトリをforkして使ってください。
https://github.com/[org]/cc-orchestration-workshop

1. 上記URLを開く
2. 右上の「Fork」ボタンを押す
3. 自分のアカウントにforkされる
4. Vercelで Import して Deploy する
```

---

#### パターン2: ビルドエラー

**定義:** `npm run dev` または `npm run build` でエラーが発生する

**対処フロー:**
1. エラーメッセージをClaude Codeに貼り付けて修正を依頼する（1回目）
2. 修正されたら再ビルドする
3. 2回修正しても解決しない場合 → 講師が直接エラーを確認して修正支援する（最大3分）
4. 3分で解決しない場合 → フォールバックリポジトリに切り替える

---

#### パターン3: 時間不足

**定義:** 予定のSTEPが時間内に完了しない

**対処フロー:**
1. 残り時間を確認し、到達可能なゴールを判断する（上記ゴール切り替え判断基準を参照）
2. デザインの簡略化（配色・レイアウトの調整を省略）
3. Git/GitHub操作を講師が代行して時間を節約する
4. 最低限ゴールに切り替え、概念理解を確実にする

**簡略化オプション:**

| 項目 | 通常 | 簡略化 |
|------|------|--------|
| ページ数 | 3-5ページ | 1ページ（トップのみ） |
| デザイン | レスポンシブ + アニメーション | テキストのみ |
| Git操作 | 参加者が自分で実行 | 講師が代行 |
| Vercel設定 | 参加者が自分で設定 | 講師が画面共有で代行 |

#### 定量データまとめ

| 指標 | 結果 | 目標 | 判定 |
|------|------|------|------|
| 完走率（公開URL取得） | __/3名 | 3/3名 | PASS / FAIL |

## C. 期限とマイルストーン

### マイルストーン一覧

| # | マイルストーン | 日程（実施日基準） | 担当 | チェックポイント | 完了判定 |
|---|-------------|-----------------|------|--------------|---------|
| MS1 | カリキュラム完成 | 実施2週間前 | 講師 | 全STEP（0-7）の手順書が完成し、通しで読んで矛盾がないこと | 手順書レビュー完了 |
| MS2 | リハーサル | 実施1週間前 | 講師 | 通し練習を1回実施、所要時間を計測、120分に収まること | タイム記録 + 問題点リスト |
| MS3 | 事前準備案内送付 | 実施5日前 | 講師 | 参加者にNotionで事前準備手順を共有 | 参加者3名全員に送付確認 |
| MS4 | 環境確認フォロー | 実施2日前 | 講師 | 参加者にDMで環境構築状況を確認、未完了者をサポート | 3名全員のClaude Code + Node.js動作確認 |
| MS5 | 本番 | 実施日 | 講師 | 当日運営、つまずきポイント記録、アンケート実施 | 講座完了 + アンケート回収 |
| MS6 | 振り返り会 | 実施1週間後 | 講師 | KPTシート回収、定量データ集計、次回開催判断 | KPT完了 + 判定結果確定 |
| MS7 | フォローアップ | 実施1ヶ月後 | 講師 | 参加者にSlackで実務適用ヒアリング | 3名全員から回答回収 |

### 詳細タスク

#### MS1: カリキュラム完成（実施2週間前）

- [ ] 全STEP（0-7）の手順書が完成している
- [ ] 各STEPの所要時間が明記されている
- [ ] 各STEPの成功条件が定義されている
- [ ] フォールバックリポジトリが作成されている
- [ ] オーケストレーション設計テンプレートが完成している
- [ ] 講師デモ台本が完成している
- [ ] 計測用Googleフォーム（事前スキル + 満足度）が作成されている

#### MS2: リハーサル（実施1週間前）

- [ ] 通し練習を1回以上実施
- [ ] 各STEPの実測時間を記録
- [ ] 120分に収まることを確認（収まらない場合は内容を調整）
- [ ] デモ（STEP2）が15分で完了することを確認
- [ ] 失敗シナリオ（ビルドエラー、pushエラー）のリカバリを練習
- [ ] フォールバックリポジトリからのfork → デプロイを確認
- [ ] Wi-Fi環境・プロジェクター接続を確認

#### MS3: 事前準備案内送付（実施5日前）

- [ ] 事前準備手順のNotionページを作成
- [ ] 参加者3名にNotionリンクを送付
- [ ] 事前スキルアンケート（Googleフォーム）のリンクを送付
- [ ] 必須ソフトウェア一覧を明記:
  - Claude Code
  - Node.js (LTS)
  - Git for Windows
  - GitHubアカウント
  - Vercelアカウント
- [ ] インストール手順をスクリーンショット付きで記載

#### MS4: 環境確認フォロー（実施2日前）

- [ ] 参加者3名にDMで以下を確認:
  - Claude Codeが起動するか
  - `node -v` でバージョンが表示されるか
  - `git --version` でバージョンが表示されるか
  - GitHubにログインできるか
  - Vercelにログインできるか
- [ ] 未完了者に対して個別サポート（画面共有 or 手順再送）
- [ ] 事前スキルアンケートの回答を確認（未回答者にリマインド）
- [ ] 当日のスキルレベル別対応方針を決定

#### MS5: 本番（実施日）

- [ ] 開始30分前に会場（またはオンライン会議室）セットアップ完了
- [ ] 講師PC環境の最終確認（デモ前チェックリスト参照）
- [ ] STEP別つまずきポイントの記録用シートを手元に用意
- [ ] 各STEPの時間管理（タイマー使用推奨）
- [ ] 終了直後にアンケートリンクを配布
- [ ] 参加者の公開URLを記録

#### MS6: 振り返り会（実施1週間後）

- [ ] アンケート結果を集計
- [ ] つまずきポイントの記録を集計（STEP別・原因分類別）
- [ ] KPTシートに記入
- [ ] 次回開催判断を実施
- [ ] 改善アクションに担当と期限を設定

#### MS7: フォローアップ（実施1ヶ月後）

- [ ] Slackで参加者3名にヒアリングメッセージを送信
- [ ] 回答を記録
- [ ] 実務適用率を算出
- [ ] 結果を踏まえてカリキュラム改善点をリストアップ

### リスクと緩和策

| リスク | 発生確率 | 影響度 | 緩和策 |
|-------|---------|-------|-------|
| 参加者の環境構築が間に合わない | 高 | 高 | MS4で2日前にフォロー、フォールバックリポジトリ用意 |
| リハーサルで120分を超過する | 中 | 高 | 各STEPのバッファ時間を設定、簡略化オプションを用意 |
| 当日のネットワーク障害 | 低 | 高 | テザリング用モバイル端末を用意 |
| Claude Code APIの障害 | 低 | 高 | フォールバックリポジトリで対応、手動コーディングは行わない |
| 参加者のスキル差が大きい | 中 | 中 | 事前アンケートでレベル把握、L3の参加者にはペアサポート役をお願い |
| アンケート回収率が低い | 低 | 中 | 講座終了直後にその場で記入してもらう（退室前に完了） |
