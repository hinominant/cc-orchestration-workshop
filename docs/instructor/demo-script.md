# 講師デモ台本 - STEP2（15分）

> 「チーム紹介ページ」を題材に、整理役・批判役・実行役の3分業 → コード生成 → デプロイまでを実演する。

---

## タイムライン

| フェーズ | 時間 | 内容 |
|---------|------|------|
| 導入 | 2分（0:00-2:00） | デモの目的説明、画面共有開始 |
| デモ | 10分（2:00-12:00） | 3役 → コード生成 → デプロイ |
| まとめ | 3分（12:00-15:00） | 要点整理、参加者への橋渡し |

## デモ（10分）

### 手順1: 整理役にサービス設計を依頼（2分）

Claude Codeを開き、以下のプロンプトを入力する。

**プロンプト全文:**

```
あなたはサービス設計の整理役です。
以下の情報を整理してください。

サービス名: チーム紹介ページ
ターゲットユーザー: 社外のお客様、採用候補者
解決したい課題: チームの顔が見えず、親しみを感じられない

以下の3点を整理してください:
1. このサービスの目的（1文で）
2. ターゲットユーザーの具体的なペルソナ（名前・立場・困りごと）
3. 提供する価値（3つ）
```

[👀] **ここで参加者に見せるポイント**: 「AIに "整理役" という役割を与えている」ことを強調する。プロンプトの冒頭を指して「ここが役割指定です」と伝える。

**講師セリフ（AIの回答を待ちながら）:**

> 「今AIが整理役として考えてくれています。ポイントは、最初に "あなたは整理役です" と役割を宣言しているところです。これがオーケストレーションの核心です。」

---

### 手順2: 批判役に弱点を指摘させる（2分）

AIの回答を確認後、続けて以下を入力する。

**プロンプト全文:**

```
次に、あなたは批判役に切り替わってください。
先ほどの整理役の設計に対して、厳しい目で弱点を指摘してください。

特に以下の観点でお願いします:
1. ターゲットの定義に穴はないか
2. 「本当にこれで課題が解決するか」の疑問点
3. 最も失敗しやすいポイント
4. 改善提案（具体的に）
```

[👀] **ここで参加者に見せるポイント**: 「同じAIに "批判役" と言い直すだけで、出力のトーンが変わる」ことを見せる。整理役の回答と批判役の回答のトーンの違いを具体的に指摘する。

**講師セリフ:**

> 「見てください。さっきは "こんな価値があります" とポジティブだったAIが、今度は "ここが甘い" と指摘しています。同じAIでも、役割を変えるだけでこれだけ違う視点が出てきます。これが3分業の力です。」

---

### 手順3: 実行役にMVP機能定義させる（2分）

**プロンプト全文:**

```
最後に、あなたは実行役に切り替わってください。
整理役の設計と批判役の指摘を踏まえて、MVPの機能を定義してください。

以下を定義してください:
1. MVP機能一覧（5つ以内、優先順位付き）
2. 各機能の1行説明
3. MVPに入れない機能と、その理由
4. 画面構成（ページ一覧）

条件:
- 静的なWebページ（HTML/CSS）で実現できる範囲に限定すること
- データベースやAPIは使わないこと
```

[👀] **ここで参加者に見せるポイント**: 「整理役と批判役の結果を踏まえて」と書いている点。AIが前のやり取りを覚えていて、設計が進化していることを見せる。

**講師セリフ:**

> 「ここまでの3ステップで、普通なら会議室で1時間かかるような設計議論が、数分で終わりました。しかも記録が全部残っています。」

### 手順4: Claude Codeでコード生成（2分）

**プロンプト全文:**

```
この設計に基づいて、Next.js（Static Export）のWebサイトを作成してください。

技術要件:
- Next.js + Tailwind CSS
- Static Export（output: 'export'）
- DB無し、API無し、フロントエンドのみ
- レスポンシブデザイン
- 日本語UI
- 画像は使わず、絵文字でメンバーのアイコンを表現

チームメンバー（ダミーデータ）:
- 佐藤 花子 / プロジェクトマネージャー / 趣味: 登山
- 鈴木 太郎 / エンジニア / 趣味: ギター
- 高橋 美咲 / デザイナー / 趣味: カフェ巡り
- 田中 健一 / マーケティング / 趣味: ランニング

ファイル構成も含めて提案し、全ファイルを作成してください。
npm run build でエラーが出ないようにしてください。
```

[👀] **ここで参加者に見せるポイント**: AIがファイルを次々と作成していく様子。「コードを1行も書いていないのに、サイトが出来上がっている」ことを強調。

**講師セリフ（AIがコードを生成中）:**

> 「今AIがコードを書いています。注目してほしいのは、私は1行もコードを書いていません。設計を言葉で伝えただけです。」

**講師セリフ:**

### 手順6: GitHub push → Vercel deploy → URL発行（1分）

```bash
git init
git add -A
git commit -m "feat: add team introduction page"
```

GitHubでリポジトリを作成し、pushする。

```bash
git remote add origin https://github.com/[ユーザー名]/team-intro-demo.git
git branch -M main
git push -u origin main
```

Vercelの画面を開き、以下の手順でデプロイする。

1. Vercel（https://vercel.com）にログイン
2. 「Add New...」→「Project」
3. GitHubリポジトリを選択
4. 「Deploy」をクリック
5. 数十秒でURLが発行される

[👀] **ここで参加者に見せるポイント**: VercelのデプロイURLを実際にスマートフォンで開いて見せる。「世界中からアクセスできる状態になった」と伝える。

**講師セリフ:**

> 「はい、公開されました。このURL、もう世界中から見られます。ここまで約10分です。皆さんもSTEP3以降で同じことをやります。」

## 失敗時のリカバリ手順

### ケース1: `npm run dev` でビルドエラー

**症状:** `Module not found` や `SyntaxError` が表示される

**対処:**
1. エラーメッセージ全文をコピーする
2. Claude Codeに以下のプロンプトを入力する:
   ```
   以下のエラーが出ました。修正してください。
   [エラーメッセージを貼り付け]
   ```
3. 修正が反映されたら再度 `npm run dev` を実行する
4. 2回修正しても解決しない場合 → フォールバックリポジトリに切り替える

**講師セリフ（エラーが起きた場合）:**

> 「エラーが出ましたが、大丈夫です。こういう時こそAIの出番です。エラーをそのまま投げれば、AIが直してくれます。」

---

### ケース2: `git push` でエラー

**症状:** `Permission denied` や `remote: Repository not found`

**対処:**
1. GitHubにログインしているか確認する
2. リポジトリが正しく作成されているか確認する
3. リモートURLを確認する:
   ```bash
   git remote -v
   ```
4. URLが間違っている場合は再設定する:
   ```bash
   git remote set-url origin https://github.com/[正しいユーザー名]/[正しいリポジトリ名].git
   ```
5. 認証エラーの場合:
   ```bash
   git config --global credential.helper store
   ```
   再度pushしてブラウザで認証する

**講師セリフ:**

> 「Gitの認証でつまずくのはよくあることです。一緒に解決しましょう。」

---

### ケース3: Vercelデプロイ失敗

**症状:** Vercelのビルドログでエラーが表示される

**対処:**
1. Vercelのビルドログを確認する
2. 「Build & Development Settings」で以下を確認:
   - Framework Preset: `Next.js`
   - Build Command: `npm run build`
   - Output Directory: `out`（Static Exportの場合）
3. `next.config.ts` に `output: 'export'` が設定されているか確認する
4. ローカルで `npm run build` が通るか確認する:
   ```bash
   npm run build
   ```
5. ローカルで通るのにVercelで失敗する場合 → Node.jsバージョンを確認し、Vercelの設定で合わせる

### ケース5: デモ用のフォールバック

事前にデモ完成版を用意しておく。万が一デモが完全に失敗した場合に切り替える。

**事前準備:**
1. デモ用リポジトリを事前に作成し、Vercelにデプロイ済みにしておく
2. URLをブックマークに保存しておく
3. ローカルにもクローン済みにしておく

**切り替え手順:**
1. 「実は事前に作ったものがあります」と言って完成版を見せる
2. 「今の手順を最後までやると、こうなります」と完成版画面を見せる
3. 「大事なのは3役の分業の考え方です」に着地する
