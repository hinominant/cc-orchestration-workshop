## 手順4: 一言でサービスアイデアを入力（3分）

**講師セリフ:**

> 「では、このAIチームに仕事を頼んでみます。たった一言です。」

Claude Codeで以下を入力:

```
チーム紹介ページを作って。メンバーは佐藤花子(PM/趣味:登山)、鈴木太郎(エンジニア/趣味:ギター)、高橋美咲(デザイナー/趣味:カフェ巡り)、田中健一(マーケティング/趣味:ランニング)の4人。各メンバーのカード表示で。
```

[👀] **参加者に見せるポイント**: 以下の3つを実況する。

1. **Sherpaが計画を立てる瞬間**: 「今、Sherpaが "何を作るか" を分解しています。ページ構成やコンポーネント構成を考えている最中です。」
2. **Artisanがコードを書く瞬間**: 「次にArtisanがコードを書き始めました。ファイルが次々と作られていくのが見えますね。」
3. **Radarが検証する瞬間**: 「最後にRadarがチェックしています。ビルドが通るか、表示が正しいか確認しています。」

**講師セリフ（エージェントが動いている間）:**

> 「注目してください。私は "チーム紹介ページを作って" と一言伝えただけです。コードは1行も書いていません。」
>
> 「AIチームが自分たちで計画を立て、コードを書き、チェックまでやっています。これがエージェントオーケストレーションです。」

**補足**: エージェントの動きが見えにくい場合は、Claude Codeのログ出力を指差しながら「今ここでSherpaが計画しています」「ここでArtisanが実装しています」と実況する。

## 想定Q&A

### Q1: 「エージェントって何ですか？」

**回答:**

> 「ざっくり言うと、 "特定の役割を持ったAI" のことです。普通のAIは何でも屋ですが、エージェントは "計画だけやる人" "コードだけ書く人" のように役割が決まっています。人間のチームでも、PMとエンジニアとテスターで役割分担しますよね。あれと同じ考え方をAIに適用したものです。」

### Q2: 「CLAUDE.mdを書き換えるとどうなりますか？」

**回答:**

> 「AIチームの動き方が変わります。たとえば "日本語で作って" を "英語で作って" に変えれば英語のサイトを作りますし、 "デザインはかわいい雰囲気で" と追記すれば、それに合わせたデザインになります。CLAUDE.mdはAIチームへの指示書なので、指示を変えれば出力が変わります。STEP4で実際に体験してもらいます。」

### Q3: 「自分の仕事でも使えますか？」

**回答:**

> 「はい、使えます。今日は "チーム紹介ページ" という題材ですが、同じ仕組みで "社内FAQ" や "イベント案内ページ" も作れます。ポイントは "何を作りたいか" を明確に言葉にすることです。AIは察してくれないので、具体的に伝えることが大事です。STEP7で、業務活用のアイデアを考える時間を取ります。」

### Q4: 「会社の機密情報を入れても大丈夫ですか？」

**回答:**

> 「Claude Codeは入力した内容をAIの学習に使いません。ただし、今日のワークショップではダミーデータを使いましょう。業務で使う場合は、所属部署のセキュリティポリシーを確認してください。機密情報をそのまま入れるのではなく、ダミーデータで構造を作ってから差し替える方法もあります。」

### Q5: 「エージェントが間違ったらどうしますか？」

**回答:**

> 「そのまま日本語で指摘すればOKです。 "ここが違うから直して" と言えば修正してくれます。デモでも見せた通り、 "ヘッダーの色を緑に変えて" と言えば修正しました。エラーが出た場合もエラーメッセージをそのまま貼り付ければ、AIが原因を特定して直してくれます。完璧を求めるのではなく、 "伝えて直してもらう" のサイクルを回すのがコツです。」

## 失敗時のリカバリ手順

### ケース1: エージェントがうまく動かない / 応答が遅い

**症状:** Claude Codeが長時間応答しない、またはエージェントが想定外の動きをする

**対処:**
1. 一度止めて、もう一度同じプロンプトを入力する
2. それでもダメな場合、「まずページの骨組みだけ作って」と範囲を絞って依頼する
3. 2回試しても解決しない場合 → フォールバックに切り替える（後述）

**講師セリフ（うまくいかない場合）:**

> 「AIも人間と同じで、たまに調子が悪い時があります。こういう時は指示を分割して小さくするのがコツです。」

---

### ケース2: `npm run dev` でビルドエラー

**症状:** `Module not found` や `SyntaxError` が表示される

**対処:**
1. エラーメッセージ全文をコピーする
2. Claude Codeに以下のプロンプトを入力する:
   ```
   以下のエラーが出ました。修正してください。
   [エラーメッセージを貼り付け]
   ```
3. 修正が反映されたら再度 `npm run dev` を実行する
4. 2回修正しても解決しない場合 → フォールバックリポジトリに切り替える

**講師セリフ（エラーが起きた場合）:**

> 「エラーが出ましたが、大丈夫です。こういう時こそAIの出番です。エラーをそのまま投げれば、AIが直してくれます。」

---

### ケース3: `git push` でエラー

**症状:** `Permission denied` や `remote: Repository not found`

**対処:**
1. GitHubにログインしているか確認する
2. リポジトリが正しく作成されているか確認する
3. リモートURLを確認する:
   ```bash
   git remote -v
   ```
4. URLが間違っている場合は再設定する:
   ```bash
   git remote set-url origin https://github.com/[正しいユーザー名]/[正しいリポジトリ名].git
   ```
5. 認証エラーの場合:
   ```bash
   git config --global credential.helper store
   ```
   再度pushしてブラウザで認証する

**講師セリフ:**

> 「Gitの認証でつまずくのはよくあることです。一緒に解決しましょう。」

---

### ケース4: Vercelデプロイ失敗

**症状:** Vercelのビルドログでエラーが表示される

**対処:**
1. Vercelのビルドログを確認する
2. 「Build & Development Settings」で以下を確認:
   - Framework Preset: `Next.js`
   - Build Command: `npm run build`
   - Output Directory: `out`（Static Exportの場合）
3. `next.config.ts` に `output: 'export'` が設定されているか確認する
4. ローカルで `npm run build` が通るか確認する
5. ローカルで通るのにVercelで失敗する場合 → Node.jsバージョンを確認し、Vercelの設定で合わせる

### ケース5: 時間が押している場合

**対処（残り時間別）:**

| 残り時間 | 対処 |

| 5分以上 | 通常通り続行 |
| 3-5分 | カスタマイズ（手順6）を省略し、デプロイまで進める |
| 2-3分 | Vercelデプロイは口頭で説明し、git pushまでをゴールにする |
| 1-2分 | ローカル表示を見せてデモ完了。「デプロイは皆さんがSTEP6で体験します」と橋渡し |
| 1分未満 | 「時間の関係でここまで。ポイントはCLAUDE.mdとエージェントの分業でした」と要点だけまとめる |

### ケース6: デモ用のフォールバック

事前にデモ完成版を用意しておく。万が一デモが完全に失敗した場合に切り替える。

**事前準備:**
1. デモ用リポジトリを事前に作成し、Vercelにデプロイ済みにしておく
2. URLをブックマークに保存しておく
3. ローカルにもクローン済みにしておく

**切り替え手順:**
1. 「実は事前に作ったものがあります」と言って完成版を見せる
2. 「今の手順を最後までやると、こうなります」と完成版画面を見せる
3. 「大事なのはCLAUDE.mdとエージェントの分業という考え方です」に着地する
